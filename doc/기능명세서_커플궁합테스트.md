# 💕 커플 궁합 테스트 — 기능명세서

> **최종 수정일**: 2026-02-02
> **버전**: v1.0
> **목표**: 커플이 각자 링크를 통해 질문에 답하고, 궁합 점수를 확인·공유할 수 있는 웹 서비스

---

## 1. 시스템 개요

### 1.1 서비스 흐름

```
A(생성자) 질문 답변
       ↓
  답변 데이터를 URL에 인코딩 → 공유 링크 생성
       ↓
B(참여자) 링크 수신 → 같은 질문에 답변
       ↓
  브라우저에서 A·B 답변 비교 → 궁합 점수 계산
       ↓
  결과 화면 출력 + (선택) 통계 서버에 결과 전송
```

### 1.2 기술 스택 (권장)

| 구분 | 기술 | 비고 |
|------|------|------|
| 프론트엔드 | React (Next.js) 또는 Vue (Nuxt) | 정적 배포 가능 |
| 백엔드 | Firebase Functions 또는 Supabase Edge Functions | 통계 API 전용, 최소 구성 |
| DB | Firebase Firestore 또는 Supabase PostgreSQL | 통계 데이터 저장용 |
| 배포 | Vercel / Netlify | 프론트 정적 호스팅 |
| URL 단축 | 자체 구현 또는 외부 API (bit.ly, dub.co) | Phase 1에서는 외부 API 활용 |
| 광고 | Google AdSense | 결과 페이지 중심 배치 |

## 2. 기능 목록 및 우선순위

### 🔴 Phase 1 — MVP (백엔드 없음, 프론트 단독)

프론트엔드만으로 동작하는 핵심 플로우.

---

#### F-01. 게임 생성 (A의 답변 플로우)

- **담당**: 프론트엔드
- **설명**: A가 질문에 하나씩 답변하는 화면
- **상세**:
  - 닉네임 입력 (필수, 10자 이내)
  - 질문을 1개씩 순차 표시 (카드 UI)
  - 답변 형식: 4지선다 (질문당 보기 4개)
  - 진행률 표시 (프로그레스 바)
  - 모든 답변 완료 후 → 공유 링크 생성 화면으로 이동

#### F-02. 공유 링크 생성

- **담당**: 프론트엔드
- **설명**: A의 답변 + 닉네임을 URL에 인코딩하여 공유 링크 생성
- **상세**:
  - 데이터 구조: `{ name: "A닉네임", answers: [1,3,2,4,...] }`
  - JSON → Base64 인코딩 → URL 쿼리 파라미터 `?d=eyJuYW1l...`
  - URL 단축 적용 (외부 API 호출)
  - **링크 복사** 버튼 (클립보드 복사)
  - **카카오톡 공유** 버튼 (카카오 SDK)
  - **일반 공유** 버튼 (Web Share API, 모바일)

#### F-03. 게임 참여 (B의 답변 플로우)

- **담당**: 프론트엔드
- **설명**: B가 공유 링크를 열고 같은 질문에 답변
- **상세**:
  - URL에서 A의 데이터 디코딩 (화면에는 노출하지 않음)
  - B도 닉네임 입력 후 동일한 질문에 답변
  - A와 동일한 질문 UI 사용
  - 답변 완료 후 → 결과 화면으로 이동

#### F-04. 궁합 점수 계산

- **담당**: 프론트엔드
- **설명**: A·B의 답변을 비교하여 점수 산출
- **상세**:
  - **총점**: 일치한 답변 수 / 전체 질문 수 × 100 (%)
  - **카테고리별 점수**: 질문을 카테고리로 분류하고 카테고리별 일치율 표시
  - 카테고리 예시: 가치관, 라이프스타일, 취향, 소통방식
  - 점수 구간별 궁합 등급 부여:
    - 90~100%: 🔥 소울메이트
    - 70~89%: 💕 찰떡궁합
    - 50~69%: 😊 노력하면 잘 맞아요
    - 30~49%: 🤔 다름이 매력
    - 0~29%: 💥 정반대 커플

#### F-05. 결과 화면

- **담당**: 프론트엔드
- **설명**: 궁합 점수와 상세 비교 결과를 보여주는 화면
- **상세**:
  - 총점 + 등급 (애니메이션 연출)
  - 카테고리별 점수 (차트 또는 게이지)
  - 질문별 A·B 답변 비교 리스트 (같으면 ✅, 다르면 ❌)
  - "가장 잘 맞는 부분" / "가장 안 맞는 부분" 하이라이트
  - **다시하기** 버튼 → 메인으로
  - **결과 공유** 버튼 (아래 F-06)

#### F-06. 결과 공유

- **담당**: 프론트엔드
- **설명**: 결과를 이미지 또는 링크로 공유
- **상세**:
  - 결과 카드를 이미지로 저장 (html2canvas 활용)
  - 카카오톡 공유 (결과 요약 + 링크)
  - 인스타그램 스토리용 이미지 비율 지원 (9:16)
  - 링크 공유 시 OG 태그 적용 (미리보기 카드)

#### F-07. 질문 데이터 관리

- **담당**: 프론트엔드
- **설명**: 질문 콘텐츠를 JSON 파일로 관리
- **상세**:
  - 최소 15문항, 권장 20~25문항
  - 질문 데이터 구조:
    ```json
    {
      "id": 1,
      "category": "lifestyle",
      "categoryLabel": "라이프스타일",
      "question": "이상적인 주말은?",
      "options": [
        { "id": 1, "text": "집에서 넷플릭스" },
        { "id": 2, "text": "카페에서 여유롭게" },
        { "id": 3, "text": "야외 활동" },
        { "id": 4, "text": "친구들과 모임" }
      ]
    }
    ```
  - 질문 순서 고정 (A·B 동일한 순서)

#### F-08. 랜딩 페이지

- **담당**: 프론트엔드
- **설명**: 서비스 소개 + 시작 버튼
- **상세**:
  - 서비스 한 줄 설명
  - "시작하기" CTA 버튼
  - 간단한 사용법 안내 (3단계: 답변 → 공유 → 결과 확인)
  - SEO 메타태그, OG 태그

#### F-09. Google AdSense 적용

- **담당**: 프론트엔드
- **설명**: 광고 배치
- **상세**:
  - **결과 페이지**: 결과 상단/하단에 배너 광고 (사용자 체류 시간 가장 긴 페이지)
  - **질문 사이 삽입 광고**: 5문항마다 or 답변 완료 직전 인터스티셜 고려
  - **랜딩 페이지 하단**: 배너 광고
  - 광고 위치가 UX를 심하게 해치지 않도록 주의
  - AdSense 승인 조건 충족: 개인정보처리방침 페이지, 충분한 콘텐츠
  - 모바일 광고 사이즈 최적화 (반응형 광고 유닛)

#### F-10. 반응형 디자인 (모바일 퍼스트)

- **담당**: 프론트엔드
- **설명**: 모바일에서 주로 사용되므로 모바일 우선 설계
- **상세**:
  - 모바일 기준 UI 먼저 설계
  - 카드 터치 영역 최소 44px
  - 하단 고정 버튼 (답변 선택 후 "다음" 등)
  - 데스크톱에서는 가운데 정렬 카드 레이아웃

### 🟡 Phase 2 — 통계 기능 (백엔드 추가)

글로벌 통계를 위한 최소한의 백엔드.

---

#### B-01. 결과 수집 API

- **담당**: 백엔드
- **설명**: 커플의 궁합 결과를 익명으로 수집
- **엔드포인트**: `POST /api/results`
- **요청 데이터**:
  ```json
  {
    "totalScore": 78,
    "categoryScores": {
      "values": 80,
      "lifestyle": 60,
      "taste": 90,
      "communication": 75
    },
    "answerComparison": [
      { "questionId": 1, "matched": true },
      { "questionId": 2, "matched": false }
    ]
  }
  ```
- **처리**:
  - 닉네임, 개인정보 일절 수집하지 않음
  - 결과만 익명 저장
  - Rate Limit 적용 (IP당 하루 10회)

#### B-02. 통계 조회 API

- **담당**: 백엔드
- **설명**: 수집된 데이터 기반 통계 제공
- **엔드포인트**: `GET /api/stats`
- **응답 데이터**:
  ```json
  {
    "totalCouples": 12847,
    "averageScore": 64.2,
    "scoreDistribution": {
      "0-29": 8,
      "30-49": 18,
      "50-69": 35,
      "70-89": 30,
      "90-100": 9
    },
    "mostDividedQuestion": {
      "questionId": 7,
      "matchRate": 23
    },
    "mostAgreedQuestion": {
      "questionId": 3,
      "matchRate": 91
    }
  }
  ```

#### F-11. 통계 결과 UI

- **담당**: 프론트엔드
- **설명**: 결과 화면에 글로벌 통계 표시
- **상세**:
  - "전체 N쌍 중 상위 XX%입니다"
  - 전체 점수 분포 그래프에서 내 위치 표시
  - "가장 의견이 갈리는 질문" / "가장 의견이 같은 질문" 표시
  - 통계 API 실패 시 해당 섹션만 숨김 (핵심 결과는 영향 없음)

### 🟢 Phase 3 — 확장 기능

성장 단계에서 추가하면 좋은 기능들.

---

#### F-12. 테마별 질문 세트

- **담당**: 프론트엔드
- **설명**: 다양한 질문 테마를 선택할 수 있게 확장
- **예시**:
  - 🔥 연인 궁합 (기본)
  - 👫 베프 궁합
  - 🏠 동거 궁합
  - 🎮 게임 취향 궁합
- 테마별 별도 질문 JSON
- 재방문 유도 효과

#### F-13. 실시간 모드

- **담당**: 프론트엔드 + 백엔드
- **설명**: 두 사람이 동시에 접속해서 실시간으로 진행
- **상세**:
  - WebSocket 또는 Firebase Realtime DB 활용
  - 대기 화면: "상대방을 기다리는 중..."
  - 동시에 한 문제씩 풀고 바로 결과 확인
  - 라이브 느낌의 연출 효과

#### F-14. 결과 히스토리 (로컬)

- **담당**: 프론트엔드
- **설명**: 이전에 했던 궁합 결과를 로컬에 저장
- **상세**:
  - LocalStorage에 최근 결과 5개 저장
  - 메인 화면에서 "지난 결과 보기" 접근
  - 용량/개인정보 이슈 없음 (로컬 한정)

## 4. 프론트 ↔ 백엔드 연동 규칙

### 4.1 독립성 원칙

- Phase 1은 **백엔드 없이 완전히 동작**해야 함
- 백엔드(Phase 2)는 **부가 기능만 담당** (통계)
- 백엔드 장애 시 핵심 기능(답변·점수·결과)은 정상 동작

### 4.2 API 연동 방식

- 프론트에서 결과 화면 렌더링 완료 후, 비동기로 `POST /api/results` 호출
- 통계 데이터는 결과 화면 로드 시 `GET /api/stats` 호출
- API 응답이 없거나 에러면 통계 섹션만 숨김 처리
- CORS 설정 필수

### 4.3 데이터 흐름도

```
[Phase 1 — 프론트 단독]

A 답변 → JSON → Base64 → URL 파라미터
                              ↓
B 링크 접속 → URL 디코딩 → A 답변 복원
                              ↓
B 답변 완료 → A·B 답변 비교 → 점수 계산 → 결과 렌더링


[Phase 2 — 백엔드 추가 시]

결과 렌더링 완료
      ↓ (비동기)
POST /api/results → DB 저장
      ↓
GET /api/stats → 통계 데이터 수신 → 통계 UI 렌더링
```

## 5. AdSense 승인 체크리스트

- [ ] 독자적인 콘텐츠 (질문·결과 텍스트)
- [ ] 개인정보처리방침 페이지 존재
- [ ] 이용약관 페이지 (권장)
- [ ] 데스크톱·모바일 정상 동작
- [ ] 페이지 로딩 속도 양호
- [ ] 충분한 페이지 수 (최소 5페이지 이상)
- [ ] 불법·성인 콘텐츠 없음
- [ ] 자체 도메인 사용 (`.com`, `.kr` 등)
- [ ] SSL 적용 (HTTPS)

## 7. 참고사항

- 결과 데이터는 영구 보관하지 않음 (URL 기반 1회성)
- 통계 DB의 데이터는 익명 집계 데이터만 보관
- 질문 콘텐츠는 정기적으로 업데이트하여 재방문 유도
- SNS 바이럴을 위한 결과 이미지 퀄리티가 핵심 성공 요인
